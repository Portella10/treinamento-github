Este relatório documenta o fluxo de trabalho Git adotado pela equipe de desenvolvimento e QA, cobrindo desde a criação de branches individuais por desenvolvedor até a promoção de código para produção. O objetivo é garantir rastreabilidade, qualidade e segurança em cada etapa do ciclo de entrega — reduzindo regressões e facilitando rollback quando necessário.

Seção 02
Fluxo de Branches — Do Dev à Produção

Nunca ouviu falar em branch? Começa aqui 👇

Imagine que o código é um documento Word compartilhado. Se todo mundo editasse o mesmo arquivo ao mesmo tempo, viraria uma bagunça. Branch é uma cópia isolada desse documento — cada pessoa trabalha na sua cópia, sem atrapalhar ninguém. Quando termina, passa por uma revisão (Pull Request), e aí sim o trabalho é juntado com o resto da equipe.

Os 4 ambientes do nosso fluxo

feature/
Sua área de trabalho
Cada tarefa ganha sua própria branch. Você cria, desenvolve e testa aqui antes de qualquer coisa.
develop
Onde o time se junta
Todas as features aprovadas chegam aqui. É o ambiente de integração — onde vemos se tudo funciona junto.
staging
Ensaio geral antes de lançar
Réplica fiel da produção. O QA testa tudo aqui. Nada avança sem aprovação explícita do QA.
main
O que está no ar
É o código em produção. Só chega aqui quem passou por todas as etapas anteriores. Tratamos como sagrado.

Fluxo de branches — do desenvolvimento à produção

O que pode travar um PR? — situações comuns

feature → develop
  • ✗Código com erro (build quebrado)
  • ✗Colega pediu ajuste na revisão
  • ✗Cobertura de testes insuficiente
  • ✗Conflito com outra branch
develop → staging
  • ✗Tarefas do sprint ainda abertas
  • ✗Testes automáticos falhando
  • ✗Changelog não atualizado
  • ✗Conflito ainda não resolvido
staging → main
  • ✗QA não deu o ok ainda
  • ✗Bug grave em aberto
  • ✗Problema de performance detectado
  • ✗Comentários do PR sem resposta
🚑
E se der um bug urgente em produção? Isso se chama hotfix. A correção é feita diretamente a partir do main, testada rapidamente, e sobe direto. Depois, obrigatoriamente, também vai para o develop — para que a correção não se perca na próxima versão.

Seção 03
Passo a Passo Prático — Ciclo Completo
01
Sincronize o repositório e crie sua branch
Sempre parta de develop atualizado. Nunca inicie uma feature a partir de uma branch desatualizada.
# Atualiza develop local git checkout develop git pull origin develop # Cria branch de feature git checkout -b feature/QA-123-login-validacao
02
Desenvolva e faça commits atômicos
Cada commit deve ter um único propósito. Use Conventional Commits. Isso facilita revisão e rollback granular.
# Adiciona somente os arquivos da tarefa git add src/components/Login.tsx # Commit com prefixo semântico git commit -m "feat(login): adiciona validação de e-mail em tempo real"
03
Abra o Pull Request para develop
Descreva o PR com: o que foi feito, como testar e screenshots / evidências. Associe ao ticket (Jira, Linear, etc.). Solicite ao menos 1 revisor.
# Envia branch ao remote git push origin feature/QA-123-login-validacao # Abra o PR via interface GitHub/GitLab # Título: feat(login): validação de e-mail — QA-123
04
CI/CD valida o PR automaticamente
O pipeline deve rodar: linting, testes unitários, build. PR bloqueado se qualquer etapa falhar. Cypress smoke tests e k6 executam no merge em develop.
05
QA valida em staging antes do release
Teste funcional, regressão, edge cases e performance. Registre evidências (screenshots, logs, relatório k6). Emita o sign-off documentado no PR ou ticket.
06
Release para main com tag semântica
Merge squash ou rebase. Crie a tag de versão. Mantenha CHANGELOG.md atualizado.
git checkout main git merge --no-ff staging git tag -a v2.4.0 -m "Release v2.4.0 — sprint 12" git push origin main --tags

Seção 04
Padrão de Commits — Conventional Commits

Mensagens de commit claras permitem geração automática de changelog e facilitam bisect de bugs. Formato: tipo(escopo): descrição curta no imperativo

Tipo Quando usar Exemplo
feat Nova funcionalidade feat(auth): implementa refresh token
fix Correção de bug fix(api): corrige 500 em timeout de sessão
test Adiciona ou corrige testes test(login): adiciona caso de senha expirada
refactor Refatoração sem mudança de comportamento refactor(utils): extrai validador para módulo próprio
chore Tarefas de manutenção (deps, config) chore: atualiza cypress para v13
docs Documentação docs: atualiza README com setup local
ci Alterações no pipeline CI/CD ci: adiciona step de smoke test no staging
⚠️
Quebre mudanças (breaking changes) devem incluir BREAKING CHANGE: no corpo do commit ou um ! após o tipo: feat(api)!: remove endpoint legado /v1/users. Isso dispara bump de versão maior automaticamente via semantic-release.
✅
Seção 05
Boas Práticas — Clean & Organized
🌿
Nomenclatura de Branches
  • Use prefixo: feature/, fix/, hotfix/, chore/
  • Inclua o ID do ticket: feature/QA-123-descricao
  • Letras minúsculas, hífens, sem espaços ou acentos
  • Nomes descritivos e curtos (máx. 50 chars)
  • Delete branches após o merge
🔍
Pull Requests
  • PR pequeno: máximo 400 linhas alteradas
  • Descrição com contexto, como testar e evidências
  • Ao menos 1 aprovação antes do merge
  • Nunca faça merge do próprio PR sem revisão
  • Resolva todos os comentários antes de mergear
🔒
Proteção de Branches
  • Bloqueie push direto em main e develop
  • Exija status checks do CI para merge
  • Exija ao menos 1 revisor aprovador
  • Ative "dismiss stale reviews" em main
  • Nunca faça force push em branches protegidas
🧪
Qualidade e QA
  • Cypress para testes E2E antes do merge em staging
  • k6 para smoke de performance a cada release
  • Cobertura mínima de 70% nos testes unitários
  • Testes de regressão antes de qualquer release
  • Documente cenários positivos, negativos e edge cases
🚑
Hotfix e Rollback
  • Hotfix sempre parte de main, não de develop
  • Após hotfix, faça merge de volta em develop
  • Rollback via git revert (nunca force reset em main)
  • Documente o incidente com causa raiz e timeline
📚
Documentação e Rastreabilidade
  • Mantenha CHANGELOG.md atualizado a cada release
  • Associe commits ao ticket de origem
  • Gere relatórios de teste com evidências no PR
  • Use tags semânticas (v1.2.3) para versionar releases
✅
Regra de ouro: Trate main como sagrado. Qualquer código que chegue a main já foi revisado, testado e aprovado. Nunca comprometa essa regra por urgência.
☑️
Seção 06
Checklist de Verificação — Antes do Merge

Use este checklist a cada Pull Request. Itens marcados com ⚠️ exigem confirmação do QA antes de prosseguir.

Developer

  • ✓ Branch criada a partir de develop atualizado
  • ✓ Commits atômicos com Conventional Commits
  • ✓ Testes unitários passando localmente
  • ✓ Sem arquivos desnecessários commitados (.env, node_modules)
  • ✓ PR com descrição, contexto e como testar
  • ✓ Reviewer(s) atribuídos
  • CI/CD passa sem erros
  • Conflitos resolvidos antes do merge

QA Engineer

  • ✓ Cenários positivos executados e aprovados
  • ✓ Cenários negativos e edge cases validados
  • ⚠ Regressão funcional executada em staging
  • ⚠ Performance validada com k6 (p95 < SLA)
  • Evidências de teste anexadas ao PR
  • Nenhum bug crítico ou bloqueante em aberto
  • Sign-off documentado no ticket
⚠️
Seção 07
Riscos, Gravidade e Mitigações
Risco Gravidade Mitigação
Push direto em main sem revisão 🔴 Alta Proteção de branch + CI obrigatório
Branches de longa duração sem sync com develop 🔴 Alta Rebase diário obrigatório; branches < 2 dias
Deploy em produção sem aprovação de QA 🔴 Alta Gate manual no pipeline + sign-off documentado
Credenciais ou segredos commitados 🔴 Alta Git hooks com detect-secrets + .gitignore revisado
PR sem cobertura de testes 🟡 Média Coverage check no CI (mín. 70%)
Merge sem resolver todos os comentários do PR 🟡 Média Regra de branch: "require conversations resolved"
Ausência de tag de versão no release 🟢 Baixa semantic-release automatizado no merge em main
Rollback sem teste em staging 🟢 Baixa Teste de rollback no runbook de incidentes
